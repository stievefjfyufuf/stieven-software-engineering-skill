# Campus Inventory End-to-End Example

This example shows how the 16 skills can be used as one traceable software engineering workflow for a campus lab equipment inventory system.

The artifacts are intentionally compact. They are not a full PRD or implementation plan; they demonstrate how IDs and handoffs should move from early discovery to deployment and later baseline change control.

## Example Flow

1. `01-inception.md` defines the problem, stakeholders, goals, assumptions, and scope.
2. `02-elicitation.md` turns stakeholder context into questions, raw needs, and ambiguities.
3. `03-specification.md` converts needs into requirements, NFRs, user stories, and acceptance criteria.
4. `04-prioritization.md` ranks scope and records decisions.
5. `05-validation-change.md` checks readiness and flags gaps.
6. `06-architecture-design.md` records components, data flow, and ADRs.
7. `07-database-api-design.md` defines data entities and API contracts.
8. `08-ui-design.md` defines screens, states, and navigation.
9. `09-issue-planning.md` creates developer-ready issues.
10. `10-implementation.md` records scoped implementation notes.
11. `11-code-review.md` records review findings and outcome.
12. `12-test-planning.md` maps tests to requirements and acceptance criteria.
13. `13-automated-testing.md` records automated test coverage and results.
14. `14-acceptance-testing.md` captures UAT evidence.
15. `15-deployment.md` records release, rollback, and baseline.
16. `16-change-request.md` demonstrates the controlled loop after baseline.
